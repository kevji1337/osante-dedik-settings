### PROBLEM 1: SSH Lockout после SSH Hardening

**Скрипт:** `02-ssh-hardening.sh`  
**Риск:** 🔴 **ВЫСОКИЙ — Потеря доступа к серверу**

**Сценарий проблемы:**
```bash
# Админ подключается по SSH с паролем
ssh root@server-ip

# Запускает скрипт SSH хардинга
./scripts/02-ssh-hardening.sh

# Скрипт применяет:
#   - PasswordAuthentication no
#   - PermitRootLogin no
#   - MaxAuthTries 3

# Перезапускает SSH
systemctl restart ssh

# РЕЗУЛЬТАТ: Текущая сессия обрывается,
# новый вход по паролю невозможен
```

**Почему это происходит:**
1. Скрипт отключает `PasswordAuthentication`
2. Если SSH ключ не был добавлен заранее — вход невозможен
3. `--no-restart` опция есть, но не используется в `quick-deploy.sh`

**Строка в скрипте:** `02-ssh-hardening.sh:120-125`
```bash
restart_ssh() {
    # ...
    systemctl restart ssh  # ← ПЕРЕЗАПУСК БЕЗ ПРОВЕРКИ КЛЮЧЕЙ
    # ...
}
```

**Решение:**
```bash
# В quick-deploy.sh добавить паузу и проверку:
./scripts/02-ssh-hardening.sh --no-restart

# Ручное применение после проверки SSH ключа:
# 1. Добавить SSH ключ в authorized_keys
# 2. Проверить вход в новой сессии
# 3. Только потом перезапустить SSH
```

**Рекомендация:**
- ⚠️ **Требовать консольный доступ (VPS console) перед запуском**
- ⚠️ **Добавить SSH ключ ДО запуска скрипта**
- ⚠️ **Использовать `--no-restart` и перезапускать вручную после проверки**

### PROBLEM 2: UFW Активация Блокирует SSH

**Скрипт:** `03-firewall-setup.sh`  
**Риск:** 🔴 **ВЫСОКИЙ — Блокировка порта 22**

**Сценарий проблемы:**
```bash
# Скрипт активирует UFW
ufw --force enable

# Но Docker ещё не настроен!
# before.rules не применены или применены некорректно

# РЕЗУЛЬТАТ: SSH порт 22 заблокирован,
# Docker networking не работает
```

**Почему это происходит:**
1. Скрипт применяет `before.rules` ДО активации UFW
2. Но если Docker ещё не установлен — правила не работают
3. UFW может заблокировать весь входящий трафик

**Строка в скрипте:** `03-firewall-setup.sh:145-155`
```bash
activate_ufw() {
    ufw --force enable  # ← АКТИВАЦИЯ БЕЗ ПРОВЕРКИ ПРАВИЛ
    # ...
}
```

**Решение:**
```bash
# В quick-deploy.sh изменить порядок:
# 1. Сначала установить Docker
# 2. Потом применять UFW

# ИЛИ использовать --dry-run для проверки:
./scripts/03-firewall-setup.sh --dry-run
ufw status verbose  # Проверить что порт 22 открыт
./scripts/03-firewall-setup.sh  # Только потом активировать
```

**Рекомендация:**
- ⚠️ **Запускать `--dry-run` перед активацией**
- ⚠️ **Проверять `ufw status verbose` перед `--force enable`**
- ⚠️ **Иметь консольный доступ на случай блокировки**

### PROBLEM 3: Sysctl Hardening Ломает Docker Networking

**Скрипт:** `05-sysctl-hardening.sh`  
**Риск:** 🟠 **СРЕДНИЙ — Docker сети перестают работать**

**Сценарий проблемы:**
```bash
# Скрипт применяет sysctl настройки
sysctl -p /etc/sysctl.d/99-hardening.conf

# Некоторые настройки конфликтуют с Docker:
#   - net.ipv4.conf.all.rp_filter = 1 (reverse path filtering)
#   - net.ipv4.conf.all.accept_redirects = 0

# РЕЗУЛЬТАТ: Контейнеры не могут общаться
# между собой или с интернетом
```

**Почему это происходит:**
1. `rp_filter = 1` может блокировать трафик от Docker bridge
2. Docker требует `ip_forward = 1` (это сохранено ✅)
3. Некоторые VPS провайдеры имеют свои sysctl ограничения

**Строка в конфиге:** `configs/sysctl.d/99-hardening.conf:23-25`
```bash
# Включить защиту от spoofing (reverse path filtering)
net.ipv4.conf.all.rp_filter = 1  # ← МОЖЕТ ЛОМАТЬ DOCKER
net.ipv4.conf.default.rp_filter = 1
```

**Решение:**
```bash
# Изменить на более мягкую настройку:
net.ipv4.conf.all.rp_filter = 2  # Loose mode
net.ipv4.conf.default.rp_filter = 2
```

**Рекомендация:**
- ⚠️ **Применять sysctl ДО установки Docker**
- ⚠️ **Использовать `rp_filter = 2` (loose mode)**
- ⚠️ **Проверять Docker networking после применения**

### PROBLEM 4: Docker Security Скрипт Ломает Coolify

**Скрипт:** `10-docker-security.sh`  
**Риск:** 🟠 **СРЕДНИЙ — Coolify деплои перестают работать**

**Сценарий проблемы:**
```bash
# Скрипт применяет daemon.json с:
{
    "userns-remap": "",    # ← ОТКЛЮЧЁНО (после исправлений)
    "userland-proxy": true # ← ВКЛЮЧЕНО (после исправлений)
}

# Но если userns-remap был включён ранее:
# - Существующие контейнеры могут не запуститься
# - Coolify может не деплоить новые приложения

# РЕЗУЛЬТАТ: Docker работает, но Coolify не может деплоить
```

**Почему это происходит:**
1. User namespace remapping меняет UID/GID в контейнерах
2. Если включить/выключить после создания контейнеров — пути к файлам не совпадают
3. Coolify кэширует конфигурацию контейнеров

**Строка в скрипте:** `10-docker-security.sh:77-88`
```bash
cat > "${daemon_json}" << 'EOF'
{
    "userland-proxy": true,    # ← ИСПРАВЛЕНО
    "userns-remap": ""         # ← ИСПРАВЛЕНО
}
EOF
```

**Решение:**
```bash
# Запускать скрипт ДО установки Coolify
# ИЛИ после остановки всех контейнеров:

docker stop $(docker ps -q)
./scripts/10-docker-security.sh
docker start $(docker ps -aq)

# Для Coolify — пересоздать приложения
```

**Рекомендация:**
- ⚠️ **Запускать ДО установки Coolify**
- ⚠️ **Если Coolify уже установлен — остановить контейнеры перед применением**
- ⚠️ **Протестировать деплой после применения**

### PROBLEM 5: Monitoring Setup Не Проверяет Порты

**Скрипт:** `08-monitoring-setup.sh`  
**Риск:** 🟡 **СРЕДНИЙ — Порты мониторинга могут конфликтовать**

**Сценарий проблемы:**
```bash
# Скрипт запускает Prometheus на порту 9090
# Но порт может быть занят другим сервисом

# Скрипт не проверяет:
# - Занят ли порт 9090
# - Занят ли порт 9093
# - Занят ли порт 9100

# РЕЗУЛЬТАТ: Prometheus не запускается,
# скрипт падает с ошибкой
```

**Почему это происходит:**
1. Скрипт создаёт systemd службы без проверки портов
2. Если порт занят — служба не стартует
3. Нет проверки перед запуском

**Строка в скрипте:** `08-monitoring-setup.sh:280-310`
```bash
start_services() {
    systemctl start prometheus  # ← БЕЗ ПРОВЕРКИ ПОРТА
    # ...
}
```

**Решение:**
```bash
# Добавить проверку портов перед запуском:
check_port_available() {
    local port=$1
    if ss -tlnp | grep -q ":${port} "; then
        log_error "Порт ${port} занят"
        return 1
    fi
}

check_port_available 9090
check_port_available 9093
check_port_available 9100
```

**Рекомендация:**
- ⚠️ **Добавить проверку портов в скрипт**
- ⚠️ **Использовать `--status` для проверки после запуска**

### PROBLEM 6: Backup Setup Требует Настройки R2

**Скрипт:** `09-backup-setup.sh`  
**Риск:** 🟡 **СРЕДНИЙ — Backup не работает без настройки**

**Сценарий проблемы:**
```bash
# Скрипт создаёт cron jobs и systemd таймеры
# Но переменные окружения не настроены:

# /etc/profile.d/restic-backup.sh содержит:
export RESTIC_REPOSITORY="r2:YOUR_BUCKET_NAME"  # ← ЗАПОЛНИТЬ!
export RESTIC_PASSWORD="YOUR_STRONG_PASSWORD"   # ← ЗАПОЛНИТЬ!

# РЕЗУЛЬТАТ: Backup скрипты падают с ошибкой,
# cron шлёт письма об ошибках
```

**Почему это происходит:**
1. Скрипт создаёт шаблон env файла с placeholder'ами
2. Cron jobs создаются ДО заполнения переменных
3. Первые запуски будут неудачными

**Строка в скрипте:** `09-backup-setup.sh:145-165`
```bash
create_env_file() {
    cat > "${env_file}" << 'EOF'
export RESTIC_REPOSITORY="r2:YOUR_BUCKET_NAME"  # ← PLACEHOLDER
EOF
}
```

**Решение:**
```bash
# В quick-deploy.sh добавить инструкцию:
echo "ВАЖНО: Заполните /etc/profile.d/restic-backup.sh"
echo "Затем выполните: restic init"
```

**Рекомендация:**
- ⚠️ **Не создавать cron jobs до заполнения переменных**
- ⚠️ **Добавить валидацию переменных перед запуском**
- ⚠️ **Использовать `--init-only` для ручной инициализации**

### PROBLEM 7: Fail2ban Может Забанить Локальный IP

**Скрипт:** `04-fail2ban-config.sh`  
**Риск:** 🟡 **НИЗКИЙ — Временная блокировка**

**Сценарий проблемы:**
```bash
# Fail2ban настроен на 5 попыток входа
# Админ пытается войти с неправильным паролем 5 раз

# Fail2ban банит IP на 1 час
# Админ теряет доступ на 1 час

# РЕЗУЛЬТАТ: Временная блокировка
```

**Почему это происходит:**
1. Скрипт настраивает `maxretry = 5`
2. Нет whitelist для доверенных IP
3. При ошибке в вводе пароля — бан

**Строка в конфиге:** `configs/fail2ban/jail.local:15-20`
```ini
[sshd]
enabled = true
maxretry = 5    # ← 5 ПОПЫТОК
bantime = 1h    # ← БАН НА 1 ЧАС
```

**Решение:**
```ini
# Добавить whitelist:
[sshd]
ignoreip = 127.0.0.1/8 ::1 192.168.1.0/24
```

**Рекомендация:**
- ⚠️ **Добавить `ignoreip` для доверенных сетей**
- ⚠️ **Иметь запасной путь доступа (console)**
- ⚠️ **Знать команду для разбана: `fail2ban-client set sshd unbanip <IP>`**
